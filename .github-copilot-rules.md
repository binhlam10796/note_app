# GitHub Copilot Rules for NotesHub Flutter Project

## Project Context
This is a Flutter notes application using:
- **Framework**: Flutter with Material Design
- **State Management**: BLoC pattern with flutter_bloc
- **Local Database**: Hive for data persistence
- **Architecture**: Clean architecture with separation of concerns

## Coding Standards

### Dart/Flutter Conventions
- Use PascalCase for class names and enums
- Use camelCase for variables, functions, and file names (except for widgets)
- Use snake_case for file names
- Use lowerCamelCase for constants that are not compile-time constants
- Use SCREAMING_SNAKE_CASE for compile-time constants
- Prefer const constructors when possible
- Use meaningful variable and function names
- Add proper documentation comments using ///

### Project-Specific Guidelines

#### File Organization
```
lib/
├── main.dart                 # App entry point
├── constants.dart           # App-wide constants
├── simple_bloc_observer.dart # BLoC debugging
├── cubits/                  # State management (BLoC)
├── models/                  # Data models with Hive annotations
└── views/                   # UI components
    ├── [screen]_view.dart   # Main screen widgets
    └── widgets/             # Reusable UI components
```

#### State Management Rules
- Use BLoC pattern for complex state management
- Create separate cubit files for different features
- Emit new states instead of mutating existing ones
- Use meaningful state names (Loading, Success, Error, etc.)
- Always handle all possible states in UI

#### Hive Integration
- All data models must extend HiveObject
- Use @HiveType() and @HiveField() annotations properly
- Run code generation after model changes: `dart run build_runner build`
- Use meaningful type IDs for Hive adapters
- Close boxes properly to prevent memory leaks

#### Widget Guidelines
- Create reusable widgets in the widgets/ folder
- Use StatelessWidget when possible
- Extract complex widgets into separate files
- Use proper widget composition over inheritance
- Follow Material Design guidelines

#### Code Quality
- Use meaningful commit messages
- Write unit tests for business logic
- Write widget tests for UI components
- Keep functions small and focused (max 20-30 lines)
- Use early returns to reduce nesting
- Prefer composition over inheritance

### Copilot Assistance Guidelines

When suggesting code:
1. **Follow BLoC Pattern**: Suggest proper cubit implementations with events and states
2. **Hive Integration**: Include proper Hive annotations and type IDs
3. **Material Design**: Use Material Design widgets and follow design principles
4. **Error Handling**: Include proper error handling and loading states
5. **Performance**: Suggest efficient widget builds and state management
6. **Testing**: Provide testable code with proper separation of concerns

### Common Patterns in This Project

#### Creating a New Feature
1. Create model in `models/` with Hive annotations
2. Create cubit in `cubits/feature_name/`
3. Create view in `views/`
4. Create reusable widgets in `views/widgets/`
5. Run code generation for Hive adapters
6. Write tests

#### Adding a New Note Field
1. Update NoteModel with new field and Hive annotation
2. Update cubit to handle the new field
3. Update UI to display/edit the new field
4. Run `dart run build_runner build`
5. Update tests

#### Creating Reusable Widgets
- Place in `views/widgets/`
- Make configurable with parameters
- Use consistent naming (Custom[WidgetName])
- Document usage with examples

### Dependencies Usage

#### flutter_bloc
- Use BlocProvider for dependency injection
- Use BlocBuilder for reactive UI
- Use BlocListener for side effects
- Use BlocConsumer when both building and listening

#### hive_flutter
- Initialize in main.dart before runApp()
- Register adapters before opening boxes
- Use meaningful box names defined in constants.dart

#### flutter_slidable
- Use for swipe actions on list items
- Configure with appropriate icons and colors
- Handle actions in parent widget

### Performance Considerations
- Use const constructors where possible
- Avoid rebuilding expensive widgets unnecessarily
- Use keys for list items that can be reordered
- Dispose controllers and close streams properly
- Use RepaintBoundary for complex widgets

### Security & Best Practices
- Validate user input
- Handle edge cases gracefully
- Use proper error boundaries
- Follow Flutter's accessibility guidelines
- Test on different screen sizes

## Example Patterns

### Cubit Implementation
```dart
// State
abstract class FeatureState {}
class FeatureInitial extends FeatureState {}
class FeatureLoading extends FeatureState {}
class FeatureSuccess extends FeatureState {
  final List<Model> items;
  FeatureSuccess(this.items);
}
class FeatureFailure extends FeatureState {
  final String message;
  FeatureFailure(this.message);
}

// Cubit
class FeatureCubit extends Cubit<FeatureState> {
  FeatureCubit() : super(FeatureInitial());
  
  void loadData() async {
    emit(FeatureLoading());
    try {
      // Load data
      emit(FeatureSuccess(data));
    } catch (e) {
      emit(FeatureFailure(e.toString()));
    }
  }
}
```

### Hive Model
```dart
import 'package:hive/hive.dart';

part 'model_name.g.dart';

@HiveType(typeId: 0)
class ModelName extends HiveObject {
  @HiveField(0)
  final String id;
  
  @HiveField(1)
  final String title;
  
  @HiveField(2)
  final DateTime createdAt;
  
  ModelName({
    required this.id,
    required this.title,
    required this.createdAt,
  });
}
```
